# ADR-001: Доменные инварианты первого production-релиза

- **Статус:** принят
- **Дата:** 2026-07-22
- **Контекст roadmap:** задача ADR-001 (этап B); блокирует TAX-001, BOOT-задачи, IMPORT-001, CAT-задачи

## Контекст

Магазин Mario Mikke готовится к первому production-релизу. Заказчик (ИП, УСН + НДС 5%)
продаёт одежду в России; продажи в СНГ отложены до стабильного запуска РФ.
Эквайринг — Т-Банк, фискализация по 54-ФЗ обязательна. Ниже зафиксированы
инварианты, на которые опирается модель данных, расчёты и интеграции. Изменение
любого из них — новый ADR.

## Решения

### 1. Валюта: только RUB

Store, регион, цены, тоталы и чеки — исключительно `rub`. Мультивалютность не
поддерживается до отдельного решения по СНГ (для каждой страны отдельно
проектируются currency/tax/payment/delivery, см. roadmap раздел 9).

### 2. НДС: 5%, цены inclusive

- Все цены каталога вводятся и хранятся **с НДС** (gross).
- Ставка НДС — **5%** (УСН + НДС 5%). Сумма налога в чеке вычисляется как
  `gross × 5 / 105`, округление — до копейки, банковское округление не
  применяется (стандартное арифметическое, согласованно во всех местах расчёта).
- В Medusa: tax region `ru` с дефолтной ставкой 5% (code `vat5`); цены
  помечаются tax-inclusive. Backend — единственный источник истины для
  цен, налогов, скидок и тоталов (правило 5 roadmap); фронтенд ничего не
  довычисляет.
- Фискальный признак для чеков (ФФД): ставка `vat5`; параметры
  налогообложения уточняются в FISCAL-001 с бухгалтером заказчика.

### 3. Размерная модель: ONE SIZE по умолчанию

~90% ассортимента — безразмерные вещи. Вариант товара обязан существовать в
Medusa даже для безразмерного товара: опция `Размер` со значением `ONE SIZE`.
Габариты вещи — атрибуты товара (metadata/measurements), не вариантные оси.
UI/API не создают комбинаций размер×цвет, которых нет в Medusa (CAT-003).

### 4. SKU и внешние идентификаторы: неизменяемые

- `SKU` варианта и `handle` товара назначаются при создании и **никогда не
  меняются** — они являются ключами сверки для повторного импорта (upsert,
  IMPORT-001) и будущей синхронизации с 1С (external ID, 1C-002).
- Скрипты импорта обязаны сверяться по `handle`/`SKU` и обновлять сущности
  на месте (upsert), сохраняя внутренние Medusa ID. Delete/recreate запрещён:
  он рвёт связи заказов и сбрасывает остатки.

### 5. РФ-first

- Регион продаж — Россия (`ru`), geo-зона доставки — `ru`.
- Обязательные поля карточки: состав, страна производства, артикул,
  инструкция по уходу (Info.md, п. 4).
- Остатки ведутся по физическим магазинам (мульти-локация, STORE/STOCK-задачи
  этапа C); до получения реестра магазинов действует одна локация
  «Основной склад».
- Один заказ резервируется в одной локации; сплит-отгрузки нет (STOCK-002).

### 6. Режимы данных витрины

- `mock`-режим — только для локальной разработки и статических превью.
- В staging и production фолбэк на демо-данные **запрещён**: недоступность
  API отображается как контролируемое состояние ошибки (FE-002, FE-007).

## Последствия

- TAX-001 реализует ставку 5% и inclusive-расчёт ровно по разделу 2.
- IMPORT-001 перестраивает импорт на upsert по правилам раздела 4.
- Бутстрап (initial-data-seed) и импорт каталога уже соответствуют разделам 1
  и 5 (BOOT-002); тесты TEST-002 закрепляют идемпотентность и стабильность ID.
- Любая интеграция (Т-Банк, СДЭК, 1С) проектируется в этих рамках; выход за
  рамки (мультивалютность, сплит-отгрузки, изменяемые SKU) — новый ADR.
